## 1. System Overview

MEGA FLOWZ is a real-time trading terminal that connects to MetaTrader 5 via Python, processes data in C++, and delivers it to a TypeScript frontend via WebSocket.

```
MetaTrader 5
    ↓ Python (connector.py)
    ↓ pybind11 bridge
C++ Engine (mega_engine)
    ↓ uWebSockets
    ↓ FlatBuffers binary frames
TypeScript Frontend
    ↓ lightweight-charts
Chart + Trading UI
```

**Key design principles:**

- Python does zero calculations — raw MT5 data only
- C++ does all data processing and message building
- FlatBuffers binary protocol — zero parse on frontend
- Frontend hot path has zero allocations on tick
- Chart is always ready before WebSocket connects — no race conditions

## 2. Architecture

### Three-Thread Backend

```
Thread 1 — Bar Pulse (100ms)
    → on_tick        — raw bid/ask per symbol
    → on_bar_update  — raw M1 OHLC numpy array

Thread 2 — Trade + History (queue-driven)
    → on_history_result  — 1000 candle numpy array
    → on_daily_open      — D1 open price
    → on_symbol_detected — detected broker symbol

Thread 3 — Accountant (500ms)
    → on_positions_update  — positions + account
    → on_connection_update — MT5 connection status
```

### Frontend Module Architecture

```
ModuleManager (orchestrator)
    ├── ConnectionManager   — WebSocket + FlatBuffers decode
    ├── ChartModule         — chart orchestrator
    │   ├── MainChart       — lightweight-charts wrapper
    │   ├── ChartDataManager— OHLCData storage + conversion
    │   ├── SeriesManager   — series create/update/destroy
    │   ├── IndicatorManager— indicator series management
    │   └── ChartDrawingModule — drawing tools
    ├── TradingModule       — orchestrator (trading.ts)
    │   ├── TradingPanel    — lot size, safe mode, risk%, TP/SL, buttons
    │   └── PositionsModal  — modal, table, inline editor, drag
    ├── WatchlistModule     — symbol price list
    ├── AlertsModule        — price alerts
    └── CalendarModule      — economic calendar
```

### Direct Wiring — Hot Path

ModuleManager holds direct references to SeriesManager and ChartDataManager.
On every tick, data goes directly — no intermediate files, no object creation.

```
Tick arrives (binary frame)
    ↓ ConnectionManager decodes
    ↓ callback fires with primitives
ModuleManager
    ↓ direct reference
SeriesManager.updateBidAsk(bid, ask)
    ↓
lightweight-charts renders
```

**Two hops only. No allocations. No GC.**

## 4. Backend — C++ Engine

### main.cpp — Entry Point

Starts Python interpreter, initializes connector bridge, connects MT5, wires all callbacks, releases GIL, starts WebSocket server.

**Critical order:**

```
1. py::scoped_interpreter start
2. connector_bridge.initialize()
3. connector_bridge.connect()         — starts 3 Python threads
4. Wire all callbacks
5. py::gil_scoped_release release
6. ws_server.start()                  — blocks forever
```

### connector_bridge.hpp — Python Bridge

Wraps Python connector via pybind11. All Python callbacks extract data, build FlatBuffers while GIL is held, then release GIL before calling C++ callbacks.

```cpp
// Pattern used everywhere
connector.attr("on_tick") = py::cpp_function(
    [this](std::string symbol, py::object tick) {
        double bid = tick.attr("bid").cast<double>();  // GIL held
        double ask = tick.attr("ask").cast<double>();  // GIL held
        py::gil_scoped_release release;                // GIL released
        if (tick_cb) tick_cb(symbol, bid, ask, time_msc); // C++ free
    }
);
```

**Changed from v1.0:**

- `BridgePositionsCallback` now takes `flatbuffers::DetachedBuffer` instead of JSON string
- `BridgeConnectionCallback` now takes `bool + string` instead of JSON string
- `checkConnection()` returns `DetachedBuffer` instead of string
- `buildPositionsJSON()` replaced by `buildPositionsFB()`

### broadcast_manager.hpp — Data Streaming

Receives callbacks from connector_bridge, resolves detected symbol to base symbol, builds FlatBuffers messages, broadcasts binary to all clients.

```
onTick()
    → resolves detected → base (e.g. BTCUSDm → BTCUSD)
    → price_update for active chart symbol
    → watchlist_update for watchlist symbols

onBarUpdate()
    → resolves detected → base
    → processM1Update() — recomputes all cached TFs
    → bar_update for active chart TF only

onPositionsUpdate()
    → receives DetachedBuffer from connector_bridge
    → broadcasts directly

onConnectionUpdate()
    → fires only on state change
    → on disconnect: saves chart state for reconnect
    → on reconnect: triggers reconnect callback
```

**Changed from v1.0:**

- All JSON string builders replaced by FBB:: calls
- `BroadcastCallback` now takes `const uint8_t*, size_t` instead of string
- `onPositionsUpdate` takes `DetachedBuffer` not string
- `onConnectionUpdate` takes `bool + string` not JSON string

### websocket_server.hpp — WebSocket Server

Runs on main thread via uWebSockets event loop. All sends are binary frames.

```
Client connects
    → flush queued messages

Client sends text command (subscribe, trade, etc.)
    → message_handler.processMessage()

Client disconnects
    → if no clients remain: clear all caches
```

**Subscribe flow:**

```
SUBSCRIBE_EURUSD_H1 received
    → detached thread → handleSubscribe()
    → cache hit:  serve immediately from symbol_cache
    → cache miss: requestDetectAndFetch() → Thread 2 queue
```

**Changed from v1.0:**

- `broadcastToAll(string)` → `broadcastToAll(uint8_t*, size_t)`
- Message queue stores `vector<uint8_t>` not string
- Ping response now FlatBuffers binary not text “pong”

### symbol_cache.hpp — Candle Cache

Per-symbol, per-timeframe candle storage. M1 updates recompute all cached timeframes in O(n_cached_tfs).

```
storeCandles()      — initial burst storage
processM1Update()   — recompute all TFs from M1 tick
getLastCandle()     — latest candle for active TF
getBaseSymbol()     — reverse lookup detected → base
getDailyOpen()      — for watchlist change % calc
```

### flatbuffer_builder.hpp — Message Builders

All message construction in one place. Replaces all JSON string building from v1.0.

```cpp
FBB::buildPriceUpdate(symbol, bid, ask, spread, time)
FBB::buildBarUpdate(symbol, timeframe, candle)
FBB::buildInitialData(symbol, timeframe, candles)
FBB::buildWatchlistUpdate(symbol, bid, ask, spread, time, change)
FBB::buildPositionsUpdate(positions, account)
FBB::buildConnectionStatus(connected, text)
FBB::buildTradeExecuted(success, direction, symbol, ...)
FBB::buildPositionClosed(success, ticket, message)
FBB::buildPositionModified(success, ticket, message)
FBB::buildError(message)
FBB::buildAutoTradingStatus(enabled, message)
FBB::buildCacheCleared(message)
FBB::buildPong()
```

Each function returns `flatbuffers::DetachedBuffer`. Caller sends via:

```cpp
ws->send(std::string_view(
    reinterpret_cast<const char*>(buf.data()),
    buf.size()
), uWS::OpCode::BINARY);
```

## 5. Python MT5 Connector

### connector.py — Three Thread Architecture

**Thread 1 — Bar Pulse (100ms)**

- Polls `symbol_info_tick()` for each active symbol
- Fires `on_tick` if tick timestamp changed
- Polls `copy_rates_from_pos(M1, 0, 1)` for live bar
- Fires `on_bar_update` with raw numpy array

**Thread 2 — Trade + History (queue-driven)**

- Processes commands from `trade_queue`
- `detect_and_fetch` — auto_detect_symbol + fetch history
- `fetch_history` — copy_rates_from_pos numpy → on_history_result
- `fetch_daily_open` — D1 open price → on_daily_open
- `execute_trade`, `close_position`, `close_all`, `modify_position`

**Thread 3 — Accountant (500ms)**

- `positions_get()` → on_positions_update
- Connection check every 5s → on_connection_update (state change only)

**Design rule — Python does zero calculations:**

- Raw numpy arrays passed directly to C++
- C++ extracts values, builds FlatBuffers
- No JSON, no formatting, no math in Python

## 6. FlatBuffers Binary Protocol

### Schema — mega_flowz.fbs

Single schema file. All messages use a union payload — one active field, zero wasted slots.

**Enums:**

```
Timeframe:    M1=0, M5=1, M15=2, H1=3, H4=4, D1=5
PositionType: Buy=0, Sell=1
```

**Candle struct** — fixed size, inline, zero allocation:

```
time:   int64
open:   float64
high:   float64
low:    float64
close:  float64
volume: int64
```

**Message union — one active payload per message:**

```
InitialData       — 1000 candle burst on subscribe
BarUpdate         — live bar every 100ms
PriceUpdate       — tick bid/ask for active chart
WatchlistUpdate   — one symbol per tick
PositionsUpdate   — positions + account every 500ms
ConnectionStatus  — MT5 connect/disconnect
TradeExecuted     — trade result
PositionClosed    — close result
PositionModified  — modify result
ErrorMsg          — error message
AutoTradingStatus — auto trading on/off
CacheCleared      — cache cleared / pong
```

**Why union instead of sparse table:**
Old approach would allocate space for all 12 fields even if only one is used.
Union = one active payload, zero wasted fields.

**Why enums instead of strings:**
`Timeframe` and `PositionType` are fixed known values.
Enum = integer compare, no heap allocation, no string comparison.
Symbol stays as string — broker symbols are dynamic (BTCUSDm, EURUSDc etc).

### Generated Files

New `flatc` v23.5.26 generates split files per table — not a single file.

```
flatc --cpp → include/generated/mega_flowz_generated.h   (DO NOT EDIT)
flatc --ts  → src/generated/mega-flowz.ts                (DO NOT EDIT)
             src/generated/mega_flowz.ts                 (DO NOT EDIT)
             src/generated/mega-flowz/                   (DO NOT EDIT)
```

`MegaFlowzDecoder.ts` imports from `./mega-flowz` — the barrel export file.
These are auto-generated by CMake build step. Never edit manually.
To regenerate: rebuild the project — CMake runs flatc automatically.

### CMake TypeScript Output Path

CMake is configured to write generated TypeScript directly to the frontend:

```cmake
set(FBS_TS_OUT "C:/Users/mega/mega_env/trading-project/src/generated")
```

If you move the project folder, update this path in `CMakeLists.txt` line 36.

### WebSocket Binary Frame

All messages sent as `uWS::OpCode::BINARY`.
Frontend sets `ws.binaryType = 'arraybuffer'`.
Frontend checks `event.data instanceof ArrayBuffer` before decoding.
Text commands (SUBSCRIBE_, TRADE_, etc.) still sent as text from frontend to backend.

Returns discriminated union:

```typescript

    | { type: 'bar_update';       data: BarUpdatePayload        }
    | { type: 'price_update';     data: PriceUpdatePayload      }

    | { type: 'connection_status';data: ConnectionStatusPayload }
    | { type: 'trade_executed';   data: TradeExecutedPayload    }
    | { type: 'position_closed';  data: PositionClosedPayload   }
    | { type: 'position_modified';data: PositionModifiedPayload }
```

### ModuleManager.ts — Orchestrator

**Chart-ready-first pattern — no race conditions:**

```typescript
public initialize(): void {
    this.initializeChartModule();
    // ... other modules ...

    // Connect ONLY after chart is ready
    this.chart.onChartReadyCallback(() => {
        this.wireDirectCallbacks();
        this.connectionManager.connect();
```

**Direct wiring — captured once, used forever:**

```typescript
private wireDirectCallbacks(): void {
    const seriesManager = this.chart.getSeriesManager();  // direct ref
    const dataManager   = this.chart.getDataManager();    // direct ref

    // Tick — active chart symbol only
    // Also passes to TradingPanel for buy/sell price display
    this.connectionManager.onTickData((symbol, bid, ask, spread, time) => {
        if (symbol !== this.connectionManager.getCurrentSymbol()) return;
        seriesManager?.updateBidAsk(bid, ask);
        this.tradingInstance?.onTick(symbol, bid, ask);
    });

    // Watchlist tick — goes to watchlist panel AND TradingModule
    // This is how inline editor gets correct prices for off-chart position symbols
    // See: Inline Editor Watchlist Tick Design section below
    this.connectionManager.onWatchlistUpdate((symbol, bid, ask, spread, time, change) => {
        this.watchlistInstance?.updatePrice(symbol, bid, change);
        this.tradingInstance?.onTick(symbol, bid, ask);
    });

    // Bar update — direct to DataManager + SeriesManager
    this.connectionManager.onBarUpdate((symbol, timeframe, candle) => {
        dataManager.updateOHLCData(ohlc);
        const latest = dataManager.getLatestUpdateForCurrentType();
        if (latest) seriesManager?.updateData(latest);
    });

    // Initial burst — direct, then clear loading state
    this.connectionManager.onCandleData((symbol, tf, candles) => {
        dataManager.addOHLCData(ohlcData);
        seriesManager?.setData(dataManager.getDataForCurrentType());
        this.chart?.setReady();
    });
}
```

### Inline Editor — Watchlist Tick Design

**The problem:**
The positions modal inline editor shows live TP/SL prices for open positions.
When a BTCUSD position is open but the chart shows EURUSD, the inline editor
needs BTCUSD prices — not EURUSD prices from the active chart tick.

**The solution — reuse watchlist tick mechanism silently:**

Backend sends `watchlist_update` for every symbol in `active_symbols`.
The watchlist panel only renders symbols in `WatchlistModule.added` Set.
These are two completely separate things.

So we can add a position symbol to `active_symbols` on the backend
without it ever appearing in the watchlist panel on the frontend.

**Flow when BTCUSD position opens and BTCUSD is NOT in user watchlist:**

```
TradingModule.updatePositions() detects new BTCUSD position
    ↓
connectionManager.sendCommand('WATCHLIST_ADD_BTCUSD')
positionTracked.add('BTCUSD')
    ↓
Backend: BTCUSD added to active_symbols
Thread 1: sends watchlist_update for BTCUSD every 100ms
    ↓
Frontend receives watchlist_update (BTCUSD, bid, ask, ...)
ModuleManager.onWatchlistUpdate:
    → watchlistInstance.updatePrice(BTCUSD) — symbol not in panel, skipped silently
    → tradingInstance.onTick(BTCUSD, bid, ask) ← inline editor gets correct price ✅
    ↓
PositionsModal.updateInlineOnTick(ask) — updates SL/TP inputs with BTCUSD price
```

**Flow when BTCUSD position closes:**

```
TradingModule.updatePositions() detects BTCUSD gone
positionTracked.has('BTCUSD') === true  ← we added it
    ↓
connectionManager.sendCommand('WATCHLIST_REMOVE_BTCUSD')
positionTracked.delete('BTCUSD')
    ↓
Backend stops BTCUSD ticks
Watchlist panel — unchanged, BTCUSD never appeared ✅
```

**Flow when BTCUSD position opens and BTCUSD IS already in user watchlist:**

```
Already getting watchlist_update ticks for BTCUSD
No WATCHLIST_ADD needed
positionTracked never adds it
On position close — not removed from watchlist ✅
```

**Key rules:**

- `positionTracked` Set lives in `trading.ts` — tracks only what TradingModule added
- Never removes symbols the user added themselves
- `WatchlistModule.added` Set is never touched by TradingModule
- Backend is dumb — it just follows WATCHLIST_ADD/REMOVE commands
- User never sees any change in the watchlist panel

**Why both `onTickData` AND `onWatchlistUpdate` call `tradingInstance.onTick()`:**

- `onTickData` fires for active chart symbol → TradingPanel buy/sell button prices
- `onWatchlistUpdate` fires for any symbol in active_symbols → inline editor correct prices
- Together they cover all cases — chart symbol AND off-chart position symbols

## 9. Old vs New — JSON to FlatBuffers

### Why FlatBuffers

|Concern             |JSON (v1.0)                            |FlatBuffers (v2.0)                      |
|--------------------|---------------------------------------|----------------------------------------|
|C++ message build   |String concat per field                |Write fixed bytes once                  |
|Payload size        |~150 bytes text                        |~60 bytes binary                        |
|JS parse            |Full string parse                      |Direct byte read                        |
|JS allocations      |1 object per message                   |0 on tick path                          |
|GC pressure         |Every tick                             |Zero on tick path                       |
|Initial 1000 candles|Parse 1000 objects — blocks main thread|Read array from buffer — zero allocation|
|CPU per tick        |5 operations                           |3 operations                            |
|GPU frame delivery  |Interrupted by GC                      |Consistent frames                       |

### GC Explanation

GC = Garbage Collector. JavaScript automatically frees unused memory.
Every `JSON.parse()` allocates an object. GC must collect it.
On 100ms tick interval: 600 parse + 600 GC cycles per minute.
GC pauses the main thread — chart stutters.
FlatBuffers = zero parse = zero GC on tick path.

### What Changed Per File

|File                   |v1.0                     |v2.0                     |
|-----------------------|-------------------------|-------------------------|
|`broadcast_manager.hpp`|builds JSON strings      |calls FBB:: builders     |
|`connector_bridge.hpp` |builds JSON strings      |builds FlatBuffers       |
|`websocket_server.hpp` |sends text frames        |sends binary frames      |
|`message_handler.hpp`  |sends JSON strings       |sends FlatBuffers        |
|`trade_handler.hpp`    |builds JSON responses    |builds FlatBuffers       |
|`main.cpp`             |JSON string callbacks    |binary callbacks         |
|`ConnectionManager.ts` |JSON.parse()             |MegaFlowzDecoder.decode()|
|`ModuleManager.ts`     |WebSocketMessage objects |raw primitives           |
|`WatchlistModule.ts`   |WebSocketMessage callback|primitive callback       |
|`trading.ts`           |one large file           |split into 3 files       |
|`trading-panel.ts`     |WebSocketMessage onTick  |primitive onTick         |
|`positions-modal.ts`   |used chart symbol ask    |uses watchlist tick ask  |

#### #P2 — Inline Editor Per-Symbol Tick via Watchlist Mechanism

**File:** `trading/trading.ts`
**Problem:**
Inline editor needs live prices for the position’s symbol, not the chart’s symbol.
Example: BTCUSD position open, chart showing EURUSD — inline editor must show BTCUSD prices.

**Solution:**
Reuse watchlist tick mechanism silently. When position opens on symbol not in user watchlist,
TradingModule sends `WATCHLIST_ADD` to backend. Backend starts ticking that symbol.
Frontend routes those ticks to `tradingInstance.onTick()` via `onWatchlistUpdate` callback.
Watchlist panel never shows the symbol — `WatchlistModule.added` Set is never modified.

**Implementation needed in trading.ts:**

```typescript
private positionTracked: Set<string> = new Set();
// connectionManager must be injected — pass via constructor or setter

public updatePositions(positions: PositionData[]): void {
    const previous = new Set(this.positions.map(p => p.symbol));
    const current  = new Set(positions.map(p => p.symbol));

    // New position symbol not in user watchlist — add silently
    current.forEach(symbol => {
        if (!previous.has(symbol) && !watchlistHas(symbol)) {
            connectionManager.sendCommand(`WATCHLIST_ADD_${symbol}`);
            this.positionTracked.add(symbol);
        }
    });

    // Closed position symbol — only remove if TradingModule added it
    previous.forEach(symbol => {
        if (!current.has(symbol) && this.positionTracked.has(symbol)) {
            connectionManager.sendCommand(`WATCHLIST_REMOVE_${symbol}`);
            this.positionTracked.delete(symbol);
        }
    });
}
```

**Note:** ModuleManager must also pass `watchlistInstance.hasSymbol(symbol)` check
or TradingModule needs access to WatchlistModule to check user watchlist.
